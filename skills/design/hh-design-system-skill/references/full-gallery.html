Sistema de Diseño de Haber Holding

Un lenguaje visual y de interacción unificado para todas las aplicaciones internas construidas en Haber Holding: dashboards, herramientas de administración, consolas operativas y flujos de captura de datos. Construido sobre la arquitectura de tokens de Material 3, expresado en la estética densa y orientada a información de HH.

Qué incluye

Cada sección de esta galería incluye ejemplos en vivo y el CSS / HTML exacto para copiar. La capa de tokens (variables CSS, con prefijo --hh-*) es la única fuente de verdad: cambia un token y todo lo que lo use se actualiza.

Fundamentos

Color, tipografía, espaciado, elevación y movimiento. La capa atómica sobre la que se construye todo lo demás.

Componentes

Botones, inputs, tablas, tarjetas, modales, etiquetas. Cada uno con variantes, estados y código listo para copiar.

Patrones

Layouts compuestos: dashboards, formularios, listas de datos con filtros. Construye nuevas pantallas en minutos.

Cómo usar este sistema

  1. Copia el bloque de tokens :root de la página Referencia de Tokens en la hoja de estilos global de tu app.
  2. Usa las clases .hh-* directamente, o envuélvelas en tu framework (React, Vue) como componentes.
  3. Nunca hardcodees colores, espaciados o tamaños de fuente. Siempre referencia un token --hh-*.
  4. Si necesitas un token que no existe, propónlo a través de la página Contribuir en lugar de crear soluciones locales.
Fundamentos

Principios de Diseño

Cuatro principios que guían cada decisión en las aplicaciones de HH.

01 · Densidad de Información

Las apps de HH sirven a operadores y analistas que necesitan ver mucho a la vez. Optimizamos por información por pulgada cuadrada, no por espacio en blanco. Los tamaños de tipografía son compactos, el padding es contenido, y el chrome es mínimo. Cada pixel se gana su lugar.

02 · Silencioso, Luego Fuerte

La interfaz es principalmente en escala de grises y silenciosa. El color se reserva para significado: azul de marca para acción, verde/amarillo/rojo para estado. Cuando algo se vuelve colorido, importa. Nunca decoramos.

03 · Filoso y Geométrico

Esquinas filosas por defecto. Sin botones redondeados, sin sombras suaves en cada tarjeta. La estética se acerca más a una Bloomberg Terminal que a un SaaS de consumo: preciso, técnico y sin sentimentalismos.

04 · Consistente en Todo HH

Un usuario que se mueve entre Compensaciones, el Dashboard de CEDIS y el Asistente de Procurement debe sentir que está dentro de un solo producto. Misma navegación, mismos botones, mismos colores de estado. La familiaridad es una característica.

Movimiento

El movimiento es funcional, no decorativo. Curvas de easing derivadas de M3, duraciones medidas en milisegundos. Lo predeterminado es rápido y sutil: las animaciones deben sentirse como que la UI está respondiendo, no actuando.

Curvas de Easing

--hh-ease-standard
cubic-bezier(0.2, 0, 0, 1)

Predeterminado para la mayoría de transiciones: entrada y salida.

--hh-ease-emphasized
cubic-bezier(0.3, 0, 0, 1)

Para cambios de estado que necesitan atención: modales, drawers.

--hh-ease-decelerate
cubic-bezier(0, 0, 0, 1)

Elementos entrando en la pantalla.

--hh-ease-accelerate
cubic-bezier(0.3, 0, 1, 1)

Elementos saliendo de la pantalla.

const hhChartTheme = { responsive: true, maintainAspectRatio: false, plugins: { legend: { labels: { color: '#999', font: { size: 10, family: 'Inter' } } }, tooltip: { backgroundColor: '#1a1a1a', borderColor: 'rgba(255,255,255,0.1)', borderWidth: 1, titleColor: '#fff', bodyColor: '#ccc', titleFont: { size: 11, family: 'Inter' }, bodyFont: { size: 11, family: 'Inter' }, } }, scales: { x: { grid: { color: 'rgba(255,255,255,0.04)' }, ticks: { color: '#999', font: { size: 10 } } }, y: { grid: { color: 'rgba(255,255,255,0.04)' }, ticks: { color: '#999', font: { size: 10 } } } } }; // Standard color sequence for series const hhVizColors = ['#41768d', '#80a4b3', '#44aa99', '#ddaa44', '#e08a3c', '#c07a8a', '#8a6fa0', '#6b8e9b'];

⚠ Crítico: Tamaño del Contenedor del Gráfico

Cuando uses responsive: true con maintainAspectRatio: false, debes envolver el canvas en un elemento padre con altura explícita en píxeles. De lo contrario, el gráfico crecerá infinitamente en cada frame de renderizado.

El bug del crecimiento infinito
Chart.js mide su padre → redimensiona → el padre crece → mide otra vez → bucle infinito. El atributo height en la etiqueta canvas se ignora bajo estos ajustes.
// ✗ WRONG — chart will grow infinitely <div class="hh-chart-box"> <canvas id="my-chart" height="200"></canvas> </div> // ✓ CORRECT — wrap in fixed-height container <div class="hh-chart-box"> <div class="hh-chart-title">Sales by Store</div> <div style="position:relative;height:220px"> <canvas id="my-chart"></canvas> </div> </div>

Alturas recomendadas: 220px para gráficos primarios, 160px para líneas de tendencia, 140px para sparklines / mini gráficos dentro de dashboards.

Cuándo usar skeleton vs. spinner

Usa skeletons

Cuando se conoce la forma del resultado (una tabla con N columnas, una grid de KPIs). Los skeletons previsualizan el layout, reduciendo el tiempo de carga percibido.

Usa un spinner

Cuando la forma del resultado es desconocida (una consulta de búsqueda, una respuesta de chat) o para acciones que interrumpen al usuario (guardar, enviar).

Contribuir

Cómo evolucionar el sistema de diseño de HH sin romperle el día a los demás.

Gobernanza

El sistema de diseño es propiedad del equipo de Transformación. Los cambios pasan por un proceso de revisión ligero para mantener consistencia entre las apps de HH.

Versionado

TipoEjemploProceso
PatchAjustar una curva de easing, corregir un typoPR directo · 1 revisor
MinorAñadir un nuevo componente o tokenRevisión de diseño + 1 revisor de dev
MajorCambiar el valor de un token, renombrar una claseRevisión completa · plan de migración · changelog

Proponer un nuevo componente

  1. Abre un ticket en ClickUp en la lista DS · Propuestas con mockup + caso de uso.
  2. Confirma que al menos 2 apps lo usarían (de lo contrario es algo único, no un componente del sistema).
  3. Usa solo tokens existentes. Si necesitas un nuevo token, propónlo por separado primero.
  4. Una vez aprobado, contribuye con el HTML/CSS + un nodo de Figma, y documéntalo en la galería.
